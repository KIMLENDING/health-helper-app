## 1. 배경 및 동기

health-helper-app은 사용자의 운동 세션을 기록하고 통계를 시각화하는 웹 애플리케이션입니다.  
기능 개발이 어느 정도 완성된 시점에서 코드를 살펴봤을 때, **동일한 문제를 각자 다른 방식으로 해결한 코드**가 여러 곳에 존재한다는 것을 발견했습니다. 특히 대시보드의 "차트 시각화" 영역은 코드가 심각하게 파편화되어 있었습니다.

> 💬 *"이 코드를 6개월 뒤에 내가 유지보수 할 수 있을까? 기억이 날까? 차트 하나를 추가하려면 몇 개의 문서를 뒤져야 할까?"*  
> 이 질문을 시작으로 프로젝트 전반의 컴포넌트 아키텍처를 재설계하는 리팩터링을 진행했습니다.

## 2. 문제 1 — 비동기 상태 처리 코드 3중 중복

### 🔴 발견한 문제

`showWeek`, `showPlans`, `showExercisePlan` 세 컴포넌트가 서버에서 데이터를 받아오는 과정에서 **로딩 스켈레톤 → 에러 UI → 빈 상태 UI → 데이터 렌더링**이라는 동일한 4단계 패턴을 각자 독립적으로 구현하고 있었습니다.

```tsx
// Before: showPlans.tsx, showWeek.tsx 등에서 동일한 패턴 반복
{isLoading ? (
  <div className="flex flex-col gap-3">
    {[1, 2, 3].map((i) => <div key={i} className="h-20 bg-gray-200 animate-pulse rounded-lg" />)}
  </div>
) : isError ? (
  <div className="text-red-500">오류가 발생했습니다...</div>
) : data && data.length > 0 ? (
  data.map((item) => <Item key={item._id} {...item} />)
) : (
  <div>데이터가 없습니다</div>
)}
```

### ✅ 해결: `AsyncStateWrapper<T>` — Generic + Render Props 패턴

**제네릭(Generic)** 과 **Render Props** 패턴을 결합해 비동기 상태를 한 곳에서 처리하는 래퍼를 만들었습니다. 동일한 조건 분기 구조 3곳이 제거되고 제네릭을 통한 타입 추론이 완성되었습니다.

```tsx
// After: showPlans.tsx
<AsyncStateWrapper
  isLoading={isLoading}
  isError={isError}
  data={data}
  loadingSkeleton={<PlanSkeleton />}
  emptyState={<EmptyPlanState />}
>
  {(plans) => plans.map((plan) => <PlanCard key={plan._id} plan={plan} />)}
</AsyncStateWrapper>
```

## 3. 문제 2 — showWeek의 조건부 렌더링 중복

### 🔴 발견한 문제

주간 달력의 각 요일을 렌더링하는 `showWeek.tsx`에서, 운동 기록(id)이 있으면 `<Link>`로, 없으면 `<div>`로 감싸는 두 경우가 **130줄 이상의 JSX 안에서 거의 동일하게 중복** 구현되어 있었습니다.

```tsx
// Before: 클래스 조합만 다른 두 블록이 반복
{day.id ? (
  <Link href={`/dashboard/detail/${day.id}`}>
    <div className={`w-10 h-10 rounded-full flex items-center justify-center
      ${day.status ? "bg-green-500 text-white" : "bg-gray-100"}
    `}>
      {/* ...내용... */}
    </div>
  </Link>
) : (
  // 위와 거의 동일한 div 블록 전체 반복...
)}
```

### ✅ 해결: `DayCell` 서브컴포넌트 + `cn()` 유틸리티

조건부 클래스 조합 로직을 하나로 통합하고 렌더링 결과를 캡슐화한 `DayCell` 서브컴포넌트를 만들었습니다. 약 80줄의 중복 코드가 컴포넌트 하나로 정리되었습니다.

```tsx
// After: DayCell 서브컴포넌트
const DayCell = ({ day }: { day: WeekDay }) => {
  const circleClass = cn(
    'w-10 h-10 rounded-full flex...',
    day.status && 'bg-green-500 text-white',
    !day.status && 'bg-gray-100'
  );

  const content = <div className={circleClass}>...</div>;

  return day.id
    ? <Link href={`/dashboard/detail/${day.id}`}>{content}</Link>
    : <div>{content}</div>;
};
```

## 4. 문제 3 — CardContainer 커스터마이징 불가

### 🔴 발견한 문제

차트 섹션에서 공통으로 쓰는 `CardContainer` 안의 Layout(`className`)이 내부에 하드코딩되어 있어, 호출하는 외부 부모 컨텍스트가 패딩/크기를 수정할 길이 없었습니다.

```tsx
// Before: 외부에서 변경 불가
const CardContainer = ({ children, title }) => (
  <Card className='basis-full'>           {/* ← 하드코딩 */}
    <CardHeader className='pb-0 px-0'>    {/* ← 하드코딩 */}
```

### ✅ 해결: `className` Props 명시적 확장 (OCP 적용)

`className`, `headerClassName`, `contentClassName` prop을 추가하여 기존 동작은 유지하면서도 필요할 때 스타일을 덮어쓸 수 있는 컴포넌트 확장성을 열어주었습니다.

## 5. 문제 4 — DataTable에 도메인 로직 하드코딩

### 🔴 발견한 문제

범용 데이터 테이블이어야 할 `DataTable` 내부에 **특정 도메인(운동 태그 및 운동 선택 로직)** 이 단단히 결합되어 다른 테이블로 활용이 불가능했습니다.

```tsx
// Before: data-table.tsx 내부. 재사용 불가!
const availableTags = ["상체", "하체", "가슴",...]; // 운동 태그 하드코딩
const useSelectedExerciseMutation = useSelectedExercise(); // 비즈니스 API 하드코딩
```

### ✅ 해결: Props 기반 외부 주입 (Inversion of Control)

도메인에 종속된 필터 요소와 클릭 시 수행할 행동(API 통신)의 책임을 콜백(`onRowSelectionChange`)으로 넘겼습니다.

```tsx
// After: 컴포넌트를 호출하는 page.tsx 등에서 주입
<DataTable
  // ...
  availableTags={['상체', '하체', '가슴', '등']}
  onRowSelectionChange={(rows) => {
    useSelectedExerciseMutation.mutate(rows);
  }}
/>
```

## 6. 문제 5 — Compound Component 적용과 Recharts 렌더링 버그 해결

### 🔴 발견한 문제

4개의 차트(`TimeByDay`, `WeightByDay` 등) 파일들이 **단순한 데이터값 외에 XAxis, Tooltip, Grid, Bar 등의 동일한 Recharts 조합 공식을 25줄씩 중복**해 깔아놓고 있었습니다.
재사용성을 높이려 Compound Component(`<BaseChart.Bar>`)를 적용했으나, 내부 래퍼(`BarItem`)로 감쌀 시 **화면에 차트 랜더링이 증발하는 기이한 버그**가 생겼습니다.

```tsx
// Error Before: 우리가 커스텀 래퍼로 감쌌으나, Recharts가 displayName을 인식 못 해 무시함.
const BarItem = (props) => <Bar {...props} />;
BaseChart.Bar = BarItem; 
```

### ❓ 원인 분석 및 ✅ 해결: 원본 매핑

Recharts는 React의 일반적인 트리 구조를 렌더링하는 대신, `child.type.displayName === 'Bar'`처럼 문자열 매칭으로 자식 종류를 수집하는 고유 렌더링 구조가 원인이었습니다.
내부 커스텀 래퍼를 걷어내고, 원본 Recharts 컴포넌트를 직접 매핑하여 컴파일을 우회했습니다.

```tsx
// After: BaseChart.tsx
BaseChart.Grid = Grid;             // 래핑 OK (Recharts가 일반 DOM 취급함)
BaseChart.Tooltip = ChartTooltip;  // 원본 매핑!
BaseChart.Bar = Bar;               // 원본 매핑!
// ...

// 사용처는 이렇게나 깔끔해졌습니다.
return (
  <BaseChart data={data} type="bar">
    <BaseChart.Grid />
    <BaseChart.XAxis dataKey="day" />
    <BaseChart.Bar dataKey="totalTime" />
  </BaseChart>
);
```

## 7. 문제 6 — 차트 스타일 제어 파편화 및 툴팁 UI 중복 (Factory Pattern)

### 🔴 발견한 문제

1. **로컬 설정 파편화와 색상 충돌**: 차트마다 로컬 설정(`WeightByPart.tsx` 안의 `chartConfig`)을 들고 있었으며, 특히 `WeightByDay`와 `TitlebyDay`가 똑같은 데이터키(`totalWeight`)를 써서 한쪽이 엉뚱한 색깔(초록색)로 렌더링되는 시각적 버그가 있었습니다.
2. **끔찍한 툴팁 HTML 중복**: 기본 툴팁만으로는 포맷(kg, 시간)이 예쁘게 안 나와 각 차트 파일마다 약 25줄의 툴팁 div 디자인(커스텀 `formatter`) 코드가 4곳이나 복사되어 있었습니다.

```tsx
// Before: 파일마다 25줄씩 존재하는 툴팁 하드코딩
formatter={(value, name, item) => (
  <div className="grid gap-1.5 w-full min-w-[150px]">
    <div className="font-semibold">{item.payload.day}</div>
    <div className="flex gap-2">
      <div className="bg-blue-400 w-2.5 h-2.5" /> 
      <span>운동시간(s)</span>
      <span>{item.payload.formetTime}</span>
    </div>
  </div>
)}
```

### ✅ 해결: SSOT 전역 통합 및 툴팁 Factory 생성

**Step 1: Single Source Of Truth (SSOT)**  
로컬 config를 지우고 오직 `lib/utils.ts` 안의 메인 `chartConfig`만 바라보게 통합했으며, 겹치는 키값을 `titleWeight`와 `totalWeight`로 분할하여 색상 충돌을 분리했습니다.

**Step 2: 팩토리 패턴 (`createTooltipFormatter`) 설계**  
직접 HTML을 쓰는 대신, `BaseChart.tsx` 내에 **필요한 옵션(props) 문자열만 넘기면 포맷 함수를 찍어내는 공장 함수**를 구현했습니다.

```tsx
// After: Factory 함수 활용 - 각 차트 코드는 단 5줄로 축약
<BaseChart.Tooltip
  formatter={createTooltipFormatter({
    titleKey: 'day',             // 헤더 텍스트 정보
    labelName: '운동시간(s)',     // 보여질 UI 라벨명
    valueKey: 'formetTime',      // 실 데이터 속성 명
    // valueSuffix: 'kg',        // (옵션 값 명시)
  })}
/>
```
특히, Factory 안에서 만들어지는 내부 반환 함수에 React DevTools 추적이 쉽도록 `TooltipContent.displayName`을 수동 배정하는 모범 기법도 적용했습니다.

## 9. 회고 및 배운 점

### "프레임워크 내부 스펙을 이해하는 것의 중요성"
이번 리팩터링에서 가장 뼈저리게 느낀 부분은 **Recharts의 `displayName` 기반 렌더링 동작 방식**이었습니다.
React의 합성 모델을 지향하여 코드를 분리했는데, 의존 라이브러리(Recharts)가 컴포넌트의 식별자를 런타임에 직접 조사해 렌더링을 차단한다는 사실을 배웠습니다. 이를 통해 서드파티 라이브러리를 추상화할 때는 라이브러리 고유의 렌더링 스펙을 철저히 검토해야 한다는 사실을 배웠습니다.

### "재사용성 = 구조적 패턴의 올바른 투입"
- **Render Props**: 제네릭을 통과시키면서도 자식이 사용할 데이터 형태를 마음대로 구성할 수 있게 하여 극한의 유연성 확보.
- **Factory Pattern (GoF)**: 단순히 컴포넌트 계층을 나누는 것에 그치지 않고, 복잡한 커스텀 로직(Tooltip)을 데이터 주도형(`props config`)으로 찍어내는 공장 함수를 만들어 코드 중복을 완전히 박멸함.
- **SSOT (Single Source Of Truth)**: 환경 설정과 키 맵핑 정보가 흩어졌을 때 일어나는 테마 오인 버그를 체감하고, 글로벌 컨텍스트에서의 정보 응집의 중요성을 깨달음.
